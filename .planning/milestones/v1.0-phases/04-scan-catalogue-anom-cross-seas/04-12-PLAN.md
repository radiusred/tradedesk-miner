<objective>
Resolve the CR-01 blocker surfaced in 04-REVIEW.md and confirmed in 04-VERIFICATION.md: every Pair-arity CROSS scan (5 of them) currently emits `Finding::ScanError("expected Pair arity (ctx.bars_pair is None)")` when invoked via `miner scan`/`engine::run_one` because `engine::run_one_with_registry` hard-codes single-leg dispatch — it builds a gap manifest for leg A only and calls `make_scan_ctx` with `bars_pair: None`. The fix is mechanical: branch on `scan.arity()` and route Pair-arity through the already-shipped `engine::gap_policy::dispatch_pair` (gap_policy.rs:206).

Equally important: tighten the regression coverage so this defect can't recur. The existing integration tests (`scan_corr_rolling.rs`, `scan_ols_rolling.rs`, `scan_lead_lag.rs`, `scan_engle_granger.rs`, `byte_identical_rerun.rs::byte_identical_rerun_cross_engle_granger`) all bypass the engine by manually constructing `ScanCtx { bars_pair: Some((a, b)), .. }` — they pin the kernel correctness but miss the facade-level wiring. `tests/two_leg_facade.rs` is currently a scaffold (it does not call `engine::run_one`); this plan converts it into the regression gate it was always meant to be. `tests/arity_preflight.rs::correct_arity_pair_scan_passes_arity_preflight` accepts ANY non-arity outcome (lines 210-213) so it would pass even when the dispatch returns ScanError — tighten it to require Finding::Result.

Out of scope (deferred to later plans, per gsd-verifier recommendation):
- WR-01..WR-08 from 04-REVIEW.md (effect_extra_keys ordering, event_window doc/impl mismatch, eom_som EOM-precedence, RawArray F64-for-UTF8 lies, etc.) — track in their own gap-closure plans or Phase 7 hardening.
- ADF reconciliation between cross/engle_granger and anom/adf — Phase 5 / HYG-01.
- Pinned-venv goldens regen — user setup recipe in 04-11-SUMMARY.md §"User Setup Required"; the 3 `#[ignore]`d tests remain ignored until the user runs that recipe.
- clippy::pedantic workspace cleanup — Phase 7.
</objective>
