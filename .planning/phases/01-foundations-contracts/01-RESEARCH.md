## Summary

Phase 1 is the foundation phase for a greenfield Rust workspace. It ships zero scans and zero readers; it ships the *constraints* that every later phase must respect. Three of those constraints — the JSONL envelope schema, the stdout discipline, and the sync-only `miner-core` invariant — are non-retrofittable, which is the whole point of putting this phase first.

The user has already made 24 locked decisions (D-01 through D-24 in CONTEXT.md). My job here is to honour those, verify current crate versions against drift since the SUMMARY.md research was written, and resolve the three open questions CONTEXT.md flagged: (1) `xtask` vs `build.rs` for schema regeneration, (2) the concrete `schemars`-1.x pattern for "base64 string with a shape", and (3) the canonical `error_code` vocabulary for Phase 1.

**Primary recommendation:** Build a virtual-manifest workspace under `crates/*` with a separate `xtask` crate, edition 2024 + MSRV 1.85 + resolver 3 (note: edition 2024 *implies* resolver 3, which is a quiet upgrade from the CONTEXT.md "resolver = 2" wording). Encode the `Finding` envelope as a `#[serde(tag = "kind")]` enum with `#[derive(Serialize, Deserialize, JsonSchema)]`; represent base64-with-shape arrays as a `RawArray { data: Base64Bytes, shape: Vec<u64>, dtype: Dtype }` struct where `Base64Bytes` is a newtype using `schemars(schema_with = ...)` to inject `contentEncoding: "base64"` / `contentMediaType: "application/octet-stream"`. Layer config via figment in the order *defaults → TOML → env → CLI*, with the CLI struct using `Option<T>` fields plus `serde(skip_serializing_if = "Option::is_none")` so unset flags do not clobber lower layers — this is the canonical figment+clap pattern and the figment docs *get the order backwards* relative to D-16's intent.

## User Constraints (from CONTEXT.md)

### Locked Decisions

All 24 decisions D-01..D-24 in `.planning/phases/01-foundations-contracts/01-CONTEXT.md` are locked and binding. Reproduced here verbatim for downstream agents (planner, plan-checker, executor):

**Finding Envelope — Raw-Array Encoding**

- **D-01: Always base64.** Every raw array embedded in a finding is encoded as little-endian `f64` bytes, base64-encoded into a JSON string. No `--raw=inline-json` flag. The Quant agent's Python decoder is the canonical one-liner: `np.frombuffer(base64.b64decode(s), dtype="<f8").reshape(shape)`. Single fast path, no encoding-mode branching in the schema. A future `miner decode-raw` CLI subcommand can be added for ad-hoc terminal inspection if it ever becomes a pain point — out of scope for Phase 1.

- **D-02: Self-contained array objects.** Each raw array is one object carrying its own bytes, shape, and dtype: `{ "data": "<base64>", "shape": [n, ...], "dtype": "f64" }`. NOT parallel `series` / `shapes` maps. Eliminates the drift bug where one map gets a key the other doesn't, and makes each array independently decodable without cross-referencing.

- **D-03: Every raw payload includes `timestamps_ms`.** When a finding ships raw arrays, the envelope always carries a `timestamps_ms` array alongside the primary numeric series (same shape/length, encoded as base64 f64 LE epoch milliseconds). The consumer can re-plot or re-test without re-querying the cache. Reason: gap-policy filtering means timestamps are NOT a regular grid derivable from `data_slice.range` — they have to be shipped explicitly. Modest size overhead (~8 bytes per bar) is acceptable.

- **D-04: Clean input/output split in the envelope.**
  - `raw.series.*` carries ONLY the input data the scan consumed (e.g., `returns`, `prices`, `timestamps_ms`). One object per input array using the D-02 shape.
  - `effect.value` is the headline scalar; `effect.extra.*` carries scan-derived outputs (e.g., Ljung-Box `lags`, `acf` array; OLS `alpha`, `beta`, `r_squared`, `residuals` array).
  - Arrays inside `effect.extra` use the same `{ data, shape, dtype }` shape as `raw.series`.
  - Consumer can distinguish "what did this scan see" from "what did this scan produce" without per-scan knowledge.

**Finding Envelope — Error Record Semantics**

- **D-05: Mid-stream errors → `kind: scan_error` finding, sweep continues.** Carries `scan_id@version`, `params`, `instrument(s)`, `side`, `timeframe`, `window`, `error_code`, `message`, `run_id`.
- **D-06: Pre-flight errors → stderr structured error + non-zero exit; stdout stays empty.**
- **D-07: Three-tier exit codes.** `0` = run completed; `1` = pre-flight or catastrophic; `2` = mixed (run completed but at least one scan_error).
- **D-08: Gap-policy outputs are findings, not errors.** Under `strict` → one `kind: gap_aborted` record per scan run, exit code 0. Under `continuous_only` → gap manifest rides inside each finding's `data_slice` field.

**Finding Envelope — Stream Framing**

- **D-09: Always emit `run_start` and `run_end` framing records.** Every invocation, every wrapper, every mode.
- **D-10: `run_id` is an always-unique ULID, time-prefixed.** 26-char Crockford-base32.
- **D-11: Rich framing payloads.**
  - `run_start`: `{ kind, run_id, started_at_utc, miner_version, code_revision, request }` (fully-resolved invocation incl. resolved params).
  - `run_end`: `{ kind, run_id, ended_at_utc, wall_clock_ms, summary }` where `summary` has `results_emitted`, `scan_errors`, `gap_aborted`, `per_scan: { "<scan_id@version>": { results, errors, gap_aborted } }`.

**Finding Envelope — Schema Artifact**

- **D-12: Publish `schemas/findings-v1.schema.json` as a checked-in public-contract artifact.**
- **D-13: Schema derived from Rust types via `schemars`.** Single source of truth.
- **D-14: One schema file per major `schema_version`.**

**Locked Envelope Fields (every finding regardless of `kind`):** `schema_version` (1 currently), `scan_id@version` (omitted on framing), `param_hash` (blake3 of resolved params), `code_revision` (git SHA or `dirty-<sha>`), `data_slice` (`range: { start_utc, end_utc }`, optional `gap_manifest_ref` under `continuous_only`), `dsr` (reserved-null in v1), `fdr_q` (reserved-null in v1).

Result findings add: `kind: "result"`, `run_id`, `produced_at_utc`, `source` (source_id, symbol, side, timeframe), `params` (resolved), `effect` (`metric`, `value`, optional `p_value`, `n`, optional `ci95`, optional `extra`), optional `raw`.

Error findings add: `kind: "scan_error"`, `run_id`, `produced_at_utc`, `error_code`, `message`, request context.

Gap-aborted findings add: `kind: "gap_aborted"`, `run_id`, `produced_at_utc`, `source`, `gap_manifest`.

Framing records carry only D-09 / D-11 fields; they do NOT carry `schema_version`.

**Claude's Discretion — Rust-Ecosystem Defaults**

- **D-15: stdout discipline mechanism.** `clippy.toml` workspace-level `disallowed-macros` banning `std::println`, `std::print`, `std::eprintln`, `std::eprint`. Exemption: only `miner-core::findings::sink` carries `#[allow(clippy::disallowed_macros)]`. Logging is `tracing` + `tracing-subscriber` writing to stderr from each binary's `main()`.
- **D-16: Config crate + format.** `figment`. TOML. Env prefix `MINER_*`. Layering: defaults < TOML file < env vars < CLI flags. Lookup: `$XDG_CONFIG_HOME/miner/miner.toml` → `./miner.toml` → `--config <path>`. Library has zero hardcoded paths.
- **D-17: Time crate.** `chrono` 0.4+, UTC always. `NaiveDate` for disk-file day keys. Never `chrono::Local`.
- **D-18: Error model.** `thiserror` typed enums in `miner-core`. `anyhow::Error` at wrapper edges. Library errors `serde::Serialize` for embedding in `scan_error` envelopes.
- **D-19: Stdout writer pattern.** Single `StdoutSink` newtype is the only `Write` against `io::stdout()`. Scans never format JSON themselves.
- **D-20: Workspace conventions.** `resolver = "2"` (NOTE: edition 2024 implies resolver 3 — see Open Risks), edition 2024, MSRV 1.85, `rust-toolchain.toml` pinned.
- **D-21: CI provider.** GitHub Actions. Four gates: workspace build, clippy with `-D warnings`, tokio-free `miner-core`, schema-sync.
- **D-22: Schema validation in CI.** A test constructs one example of each envelope `kind`, serialises, validates against `findings-v1.schema.json` via `jsonschema` crate.
- **D-23: `miner-bench` scaffolding for Phase 1.** Empty crate; placeholder respects stdout discipline.
- **D-24: ULID crate.** `ulid` (most-popular Rust ULID impl). Seeded-RNG inside generation is acceptable in v1; deterministic-ULID belongs to Phase 5.

### Claude's Discretion

Everything inside D-15 through D-24 is Claude's discretion — the user explicitly delegated Rust-ecosystem choice to "the most pragmatic and/or popular options Rust community members would recognise." This research recommends keeping the D-15..D-24 defaults exactly as written, with two surgical refinements documented later in this document:

### Pattern 1: Discriminated-Union Envelope via `#[serde(tag = "kind")]`

**What:** Encode the `Finding` envelope as a serde-tagged enum so a single Rust type produces every JSON variant.

**When to use:** This is the standard Rust idiom for tagged unions, and JSON Schema's `oneOf` translation handles it cleanly.

**Example:**

```rust
// crates/miner-core/src/findings/mod.rs
use serde::{Serialize, Deserialize};
use schemars::JsonSchema;

#[derive(Debug, Clone, Serialize, Deserialize, JsonSchema)]
#[serde(tag = "kind", rename_all = "snake_case")]
pub enum Finding {
    RunStart(RunStart),
    Result(ResultFinding),
    ScanError(ScanErrorFinding),
    GapAborted(GapAbortedFinding),
    RunEnd(RunEnd),
}

#[derive(Debug, Clone, Serialize, Deserialize, JsonSchema)]
pub struct RunStart {
    pub run_id: RunId,
    pub started_at_utc: chrono::DateTime<chrono::Utc>,
    pub miner_version: String,
    pub code_revision: String,
    pub request: serde_json::Value,  // see Open Questions on RawValue alternative
}
// ... and so on for the other four variants.
```

Source: serde docs on enum representations + schemars docs (tagged enums become `oneOf` with a `const` for `kind`).

### Pattern 2: Newtype Around a String for Base64-Encoded Bytes

**What:** A `Base64Bytes(Vec<u8>)` newtype carries (a) the raw bytes, (b) custom serde impls that emit/parse base64, (c) a `JsonSchema` impl that emits `{"type":"string","contentEncoding":"base64","contentMediaType":"application/octet-stream"}`.

**When to use:** Whenever a struct field holds binary data that must round-trip as a base64 string in JSON and be schema-described.

**Example (concrete shape — to be implemented in Phase 1):**

```rust
// crates/miner-core/src/findings/base64_bytes.rs
use base64::{Engine, engine::general_purpose::STANDARD};
use serde::{Serialize, Deserialize, Serializer, Deserializer};
use serde::de::Error as _;
use schemars::{Schema, SchemaGenerator, JsonSchema};

#[derive(Debug, Clone, PartialEq)]
pub struct Base64Bytes(pub Vec<u8>);

impl Serialize for Base64Bytes {
    fn serialize<S: Serializer>(&self, s: S) -> Result<S::Ok, S::Error> {
        s.serialize_str(&STANDARD.encode(&self.0))
    }
}

impl<'de> Deserialize<'de> for Base64Bytes {
    fn deserialize<D: Deserializer<'de>>(d: D) -> Result<Self, D::Error> {
        let s = String::deserialize(d)?;
        STANDARD.decode(&s).map(Base64Bytes).map_err(D::Error::custom)
    }
}

impl JsonSchema for Base64Bytes {
    fn schema_name() -> std::borrow::Cow<'static, str> { "Base64Bytes".into() }
    fn schema_id() -> std::borrow::Cow<'static, str> { "miner_core::findings::Base64Bytes".into() }
    fn json_schema(_: &mut SchemaGenerator) -> Schema {
        // schemars 1.x: Schema wraps serde_json::Value. Build the object directly.
        serde_json::json!({
            "type": "string",
            "contentEncoding": "base64",
            "contentMediaType": "application/octet-stream",
            "description": "Little-endian f64 bytes, base64-encoded"
        }).try_into().expect("valid schema fragment")
    }
}
```

`RawArray` then composes naturally:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, JsonSchema)]
pub struct RawArray {
    pub data: Base64Bytes,
    pub shape: Vec<u64>,
    pub dtype: Dtype,             // an enum: F64, F32, I64, ... — only F64 used in v1
}
```

Source for the `Base64Bytes` recipe: schemars 1.x changelog ("Schema now wraps serde_json::Value"), JSON Schema 2020-12 `contentEncoding` / `contentMediaType` definitions. The base64 crate's `STANDARD` engine is the current 0.22+ idiom.

### Pattern 3: FindingSink trait + StdoutSink newtype

**What:** Per D-19, all envelope serialisation flows through `FindingSink::write_envelope`. Phase 1 ships the trait and one impl, even though no scans exist yet — wrappers can already test it by writing a `Finding::RunStart`.

**Example:**

```rust
// crates/miner-core/src/findings/sink.rs
use std::io::{BufWriter, Stdout, Write};
use crate::findings::Finding;
use crate::error::MinerError;

pub trait FindingSink: Send {
    fn write_envelope(&mut self, finding: &Finding) -> Result<(), MinerError>;
    fn flush(&mut self) -> Result<(), MinerError>;
}

pub struct StdoutSink {
    writer: BufWriter<Stdout>,
}

impl StdoutSink {
    pub fn new() -> Self { Self { writer: BufWriter::new(std::io::stdout()) } }
}

impl FindingSink for StdoutSink {
    fn write_envelope(&mut self, finding: &Finding) -> Result<(), MinerError> {
        serde_json::to_writer(&mut self.writer, finding)
            .map_err(MinerError::Serialize)?;
        self.writer.write_all(b"\n").map_err(MinerError::Io)?;
        // line-flush so a panic loses at most the in-flight envelope (PITFALLS #4)
        self.writer.flush().map_err(MinerError::Io)?;
        Ok(())
    }
    fn flush(&mut self) -> Result<(), MinerError> {
        self.writer.flush().map_err(MinerError::Io)
    }
}
```

`StdoutSink` is the *only* type in the workspace that touches `io::stdout()`. The lint rule (clippy.toml) makes that mechanically true.

### Pattern 4: figment Builder Factory in the Library

**What:** `miner-core::config` exposes a builder function that takes (a) the CLI-derived overlay struct (typed, `Option<T>` fields) and (b) optional explicit-path override and assembles a figment. The wrappers call this; the library carries no environment-aware code paths.

**Example:**

```rust
// crates/miner-core/src/config/mod.rs
use figment::{Figment, providers::{Format, Toml, Env, Serialized}};
use serde::{Serialize, Deserialize};
use std::path::PathBuf;

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct MinerConfig {
    pub cache_root: PathBuf,
    pub bar_cache_root: PathBuf,
    pub output: OutputDest,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum OutputDest {
    Stdout,
    File(PathBuf),
}

#[derive(Debug, Clone, Serialize, Deserialize, Default)]
pub struct CliOverrides {
    #[serde(skip_serializing_if = "Option::is_none")]
    pub cache_root: Option<PathBuf>,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub bar_cache_root: Option<PathBuf>,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub output: Option<OutputDest>,
}

pub fn build(cfg_file: Option<&std::path::Path>, cli: CliOverrides)
    -> Result<MinerConfig, figment::Error>
{
    let mut fig = Figment::new();
    // (no defaults inside the library; missing field = error)
    if let Some(path) = cfg_file {
        fig = fig.merge(Toml::file(path));
    }
    fig = fig
        .merge(Env::prefixed("MINER_"))         // MINER_CACHE_ROOT etc.
        .merge(Serialized::defaults(cli));       // CLI wins
    fig.extract()
}
```

Wrapper resolves XDG path + explicit `--config` then calls `build`. Order: TOML < env < CLI. The `skip_serializing_if` makes CLI-unset fields invisible to figment, so env/TOML survive.

### Pattern 5: tracing-subscriber initialised per binary, ALWAYS to stderr

**What:** Each binary's `main()` initialises a `tracing_subscriber` that writes to stderr with `EnvFilter` from `RUST_LOG`. The library does not depend on `tracing-subscriber`; it only emits via the `tracing` macro crate.

**Example:**

```rust
// crates/miner-cli/src/main.rs
use tracing_subscriber::{fmt, EnvFilter};

fn main() -> anyhow::Result<()> {
    tracing_subscriber::fmt()
        .with_writer(std::io::stderr)
        .with_env_filter(EnvFilter::try_from_default_env()
            .unwrap_or_else(|_| EnvFilter::new("info")))
        .init();

    // ... clap, figment, dispatch ...
}
```

Source: tracing-subscriber docs; verified standard idiom.

### Anti-Patterns to Avoid

- **`#[serde(flatten)]` on the locked-envelope-fields struct** — feels DRY, but `schemars` historically generates an "additionalProperties: true" schema when flatten is used, which weakens the schema as a contract. Either inline the fields in each variant, or use an intermediate struct with explicit re-serialisation. (Schemars 1.x improves this — verify during the spike.)
- **Using `HashMap` for the `effect.extra` or `raw.series` shapes** — non-deterministic JSON output ordering, breaks the golden-file gate Phase 1 lays groundwork for (OUT-03). Use `BTreeMap` or an `IndexMap` with a deterministic insertion order.
- **`println!` anywhere in `miner-core`** — D-15. Even in tests, prefer `tracing::info!` (which the test harness routes appropriately) so the lint stays clean.
- **`async fn` or `#[tokio::main]` anywhere in `miner-core`** — D-20 + the tokio-free CI gate. Wrappers may use them; the core may not.
- **Hand-writing `schemas/findings-v1.schema.json`** — D-13 makes this a generated artifact. Hand-edits will fail the CI sync gate.
- **`figment` CLI layered first** — the figment docs show this order; for our precedence (CLI > env > TOML), CLI must be merged LAST. See Config Precedence Mechanics.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON Schema generation from types | A `derive` macro of your own | `schemars::JsonSchema` | A community-standard derive that handles 95% of cases; only escape into custom impls for cases like `Base64Bytes`. |
| Hierarchical config (defaults / file / env / flags) | A bespoke loader | `figment` | Provenance tracking, profile support, well-tested merge semantics. |
| stdout/stderr lint enforcement | A custom build-script grep | `clippy.toml` `disallowed-macros` | Built into clippy, integrates with `cargo clippy`, easy to add per-module exemptions. |
| Structured logging | `println!` to stderr | `tracing` + `tracing-subscriber` | Spans, fields, filtering, future structured-output options (JSON logs) for free. |
| ULID generation | A `rand` + `time` hand-roll | `ulid` crate | Crockford encoding correctness, monotonic counter on collisions, well-tested. |
| Runtime JSON Schema validation in tests | Hand-rolled validators | `jsonschema` crate | Draft 2020-12 support, performance, structured output. |
| TOML parsing | `toml` directly | `figment::providers::Toml` | Already a transitive dep; one less direct dep. |
| Base64 encoding/decoding | Hand-rolled tables | `base64` crate (0.22+ `STANDARD` engine) | Mature, fast, the new engine API replaced the deprecated free functions. |
| `param_hash` over resolved params | SHA2 / SHA3 by hand | `blake3` crate | Fast, deterministic, designed for content addressing. |
| Cross-platform `$XDG_CONFIG_HOME` resolution | `env::var("XDG_CONFIG_HOME")` plus fallback heuristics | `directories` crate | Wins the Linux-vs-macOS-vs-Windows lookup; one less foot-gun. |

**Key insight:** Phase 1 is entirely about *establishing constraints*. Hand-rolling anything in this list re-opens a question the Rust ecosystem has already settled, and the constraint loses its force because the implementation will diverge from what reviewers expect.

## Common Pitfalls

These are the Phase-1-specific pitfalls. The full project pitfalls catalogue is in `.planning/research/PITFALLS.md`. Phase 1 closes #2 (multiple-testing — schema fields reserved), #4 (stdout discipline), #5 (async contamination — workspace), and #6 (schema breakage). #11 (hardcoded paths) is also closed by D-16. #1 and #7 (look-ahead bias, naive aggregation) are reserved for Phase 2/3; the envelope must accommodate their outputs (it does — `data_slice`, `effect.extra`).

### Pitfall 1: The figment + clap precedence inversion

**What goes wrong:** The figment documentation example shows `Serialized::defaults(Config::parse())` merged FIRST (so file/env overrides CLI). But D-16 requires CLI > env > TOML — the OPPOSITE order.

**Why it happens:** A naive copy of the figment docs example. Compounding the problem: if you reverse the order naïvely (`Serialized::defaults(Config::parse())` merged LAST), clap's *default* values for unspecified flags will overwrite env/TOML — because clap fills `String` fields with their default, and figment can't tell "I set this to the default" from "I didn't set this".

**How to avoid:** Use the steezeburger.com pattern — make the CLI overlay struct's fields `Option<T>`, derive `Serialize` on it with `#[serde(skip_serializing_if = "Option::is_none")]` on every field, and merge it last. See Pattern 4 above and Config Precedence Mechanics below for the exact shape.

**Warning signs:** A test that sets `MINER_CACHE_ROOT=/foo` and does not pass `--cache-root` still ends up resolving `cache_root` to whatever clap's default was (typically empty path or panics).

### Pitfall 2: schemars 1.x Schema = serde_json::Value

**What goes wrong:** Code written against `schemars 0.8` (which has a typed `SchemaObject` builder) does not compile against 1.x (which collapses everything down to `Schema(serde_json::Value)`). The custom `JsonSchema` impl you write for `Base64Bytes` looks superficially the same in both versions but the assembly is very different.

**Why it happens:** Major version transition between 0.8 → 1.0; older Stack Overflow / blog posts show the 0.8 pattern.

**How to avoid:** Write all custom impls against schemars 1.x docs, not blog posts. The spike task in Phase 1 must compile and run `cargo xtask gen-schema` against the chosen crate version before any other code is written against the envelope.

**Warning signs:** Compile errors from `schemars::schema::SchemaObject` (1.x removed/relocated that type).

### Pitfall 3: thiserror + serde::Serialize derive does not compose for IO errors

**What goes wrong:** `MinerError` needs `serde::Serialize` (per D-18, "library errors implement `serde::Serialize` so they can be embedded inside `kind: scan_error` envelopes via D-05"). But if `MinerError::Io(#[from] std::io::Error)` is in the enum, `#[derive(Serialize)]` *will not compile* because `std::io::Error` does not implement `Serialize`.

**Why it happens:** It's a real ecosystem hole that catches people on first contact.

**How to avoid:** Hand-implement `Serialize` for `MinerError` to call `serializer.serialize_str(&self.to_string())` (i.e., serialise as the `Display` string). The `thiserror`-derived `Display` impl already gives the right human-readable message; serialising as a string is what the `error_code` + `message` envelope contract wants anyway. Alternatively, factor the error into two layers: an internal `MinerError` with `#[from] io::Error` and no `Serialize`, and a wire-level `WireError { code: String, message: String }` that has `Serialize`. Lean toward the latter — cleaner separation, and it forces the `error_code` vocabulary discipline (see error_code Vocabulary section).

**Warning signs:** A compile error like `the trait Serialize is not implemented for std::io::Error` when first adding `#[derive(Serialize)]` to the error enum.

### Pitfall 4: Forgetting to flush on EVERY emit

**What goes wrong:** `BufWriter` is the right choice for stdout performance, but if `flush()` is not called per-envelope, a panic mid-sweep loses all buffered findings — the agent sees a truncated stream and cannot tell whether the run completed cleanly.

**Why it happens:** "I'll flush at the end" is the default pattern.

**How to avoid:** `write_envelope` ends with `self.writer.flush()?;` — line-buffered semantics for findings, period. See Pattern 3.

**Warning signs:** Tests pass; a SIGINT or panic mid-run loses the last few findings; `run_end` is missing from the stream.

### Pitfall 5: Edition 2024 implicitly upgrading resolver to v3

**What goes wrong:** CONTEXT.md D-20 says `resolver = "2"`. But edition 2024 *implies* `resolver = "3"`. If the workspace sets `edition = "2024"` and `resolver = "2"`, cargo issues no warning but also silently does not enable the MSRV-aware dependency selection. This is harmless on day 1 but produces confusing dep-resolution behaviour later.

**Why it happens:** Resolver versions ratchet with editions; the change is mostly invisible.

**How to avoid:** Write `resolver = "3"` in the workspace Cargo.toml. Document the divergence from D-20 in the PLAN's Risks section as a non-substantive correction (the user's intent was "use the current resolver"; resolver 3 is the current resolver).

**Warning signs:** A `cargo update` that picks a dep version that requires a newer compiler than 1.85 — resolver 3 would have rejected this.

### Pitfall 6: `directories` vs `dirs` — two crates, same problem

**What goes wrong:** Both crates resolve `$XDG_CONFIG_HOME` cross-platform but with different APIs. `directories` is the slightly more idiomatic choice (returns typed `ProjectDirs`), `dirs` is simpler (just functions). They DO NOT interoperate.

**How to avoid:** Pick one. This research recommends `directories` for the typed-newtype ergonomics. Phase 1 plan should not import both.

### Example 4: `build.rs` for code_revision injection

```rust
// crates/miner-core/build.rs
use std::process::Command;

fn main() {
    let sha = Command::new("git").args(["rev-parse", "HEAD"]).output()
        .ok().and_then(|o| if o.status.success() { String::from_utf8(o.stdout).ok() } else { None })
        .map(|s| s.trim().to_string())
        .unwrap_or_else(|| "unknown".to_string());

    // Detect dirty tree
    let dirty = Command::new("git").args(["diff", "--quiet"]).status()
        .map(|s| !s.success()).unwrap_or(false);

    let rev = if dirty { format!("dirty-{}", sha) } else { sha };
    println!("cargo:rustc-env=MINER_CODE_REVISION={}", rev);
    // Re-run only if .git/HEAD changes:
    println!("cargo:rerun-if-changed=.git/HEAD");
}
```

Then in `miner-core/src/lib.rs`: `pub const CODE_REVISION: &str = env!("MINER_CODE_REVISION");`.

## Open Questions

The CONTEXT.md flagged three open questions for the research phase to resolve. Each is now answered below.

1. **xtask vs build.rs for schema regeneration.**
   - **What we know:** `build.rs` runs at every build and would slow `cargo check` / `cargo build` for every contributor. `xtask` is opt-in (developers / CI invoke it explicitly) and is the matklad pattern that cargo itself uses for its own dev tooling.
   - **What's unclear:** Whether the CI gate should run `cargo xtask gen-schema && git diff --exit-code schemas/` (the recommended pattern) or whether the schema regen should hide behind a feature flag in `miner-core` that `cargo run --features regen-schema` triggers.
   - **Recommendation:** **`xtask`**. Reasons: (a) idiomatic, (b) zero impact on build time, (c) keeps the regen logic out of `miner-core`'s dep surface (no `schemars` need leak into runtime dependencies — it stays a dev-dependency of `xtask` only — *correction*: it has to be a real dep of `miner-core` because the derived `JsonSchema` impls live on the production types, but `xtask` is the *invoker*, not the *site* of regeneration. So: `schemars` is a non-optional dep of `miner-core`, and `xtask` depends on `miner-core` and calls `schemars::schema_for!`.). The `build.rs` recipe is reserved for `CODE_REVISION` injection only (Example 4).

2. **How to express base64-with-shape inside a `schemars`-derived JSON Schema.**
   - **What we know:** `schemars` 1.x removed the typed `SchemaObject` builder; `Schema` is now a thin wrapper around `serde_json::Value`. Manual `JsonSchema` impls are still supported. The `#[schemars(schema_with = "fn")]` attribute is supported and is the lighter-weight option.
   - **What's unclear:** Whether the cleaner pattern is (a) a `Base64Bytes` newtype with a manual `JsonSchema` impl (Pattern 2 above) so the *field type* tells schemars what to do, or (b) raw `Vec<u8>` plus `#[schemars(schema_with = "base64_schema")]` per field. Pattern (a) wins because the newtype also gives you typed `Serialize`/`Deserialize` (decoding bytes from the base64 string belongs on the type, not the field). Pattern (b) leaves the deserialiser doing string-of-bytes wrangling per field.
   - **Recommendation:** **Newtype + manual `JsonSchema` impl** (Pattern 2). A Phase 1 spike task must compile this, run `cargo xtask gen-schema`, and confirm the output contains `"contentEncoding": "base64"` against schemars 1.0.5+ before any envelope variant is fleshed out.

3. **Exact set of `error_code` strings for D-05 / D-06.**
   - Resolved in detail in the error_code Vocabulary section below.

## Schema Derivation Strategy

**Decision: `xtask`.** Open Question 1 from CONTEXT.md is answered.

### Why `xtask`

1. **Build-time cost.** `build.rs` runs on every `cargo build`, every `cargo check`, every `rust-analyzer` background analysis. Regenerating a JSON Schema is cheap individually but multiplied across every IDE save, it adds up. `xtask` runs only when invoked.
2. **Reverse dep direction is cleaner.** With `build.rs` inside `miner-core`, the build script of the *library* writes a file in the *workspace*, which crosses the crate boundary in an awkward direction. With `xtask`, an external binary depends on `miner-core` and pulls schemas out — natural data flow.
3. **Idiomatic.** matklad's "Large Rust Workspaces" post — cited in `/.planning/research/SUMMARY.md` Source list — and cargo itself use this pattern.
4. **The two CI gates are easy.** Sync gate: `cargo run -p xtask -- gen-schema && git diff --exit-code schemas/`. Validation gate: `cargo test -p miner-core --test schema_roundtrip` (which reads the file from disk and validates samples).

### What `xtask gen-schema` does

```rust
// xtask/src/main.rs (sketch)
use clap::{Parser, Subcommand};
use miner_core::findings::Finding;

#[derive(Parser)]
struct Cli { #[command(subcommand)] cmd: Cmd }

#[derive(Subcommand)]
enum Cmd {
    GenSchema { #[arg(default_value = "schemas/findings-v1.schema.json")] out: std::path::PathBuf },
}

fn main() -> anyhow::Result<()> {
    let cli = Cli::parse();
    match cli.cmd {
        Cmd::GenSchema { out } => {
            let schema = schemars::schema_for!(Finding);
            let pretty = serde_json::to_string_pretty(&schema)?;
            std::fs::write(&out, format!("{}\n", pretty))?;
            eprintln!("wrote {}", out.display());  // xtask is a dev tool; eprintln in xtask is fine — clippy.toml exemption applies only to miner-core if you scope it; alternatively, use println! since xtask never produces findings. SAFER: scope the lint to crates/miner-* and exempt xtask entirely.
            Ok(())
        }
    }
}
```

**Lint scoping nuance:** `clippy.toml` at the workspace root applies to *every* crate by default, including `xtask`. The simplest fix: keep the workspace clippy.toml as-is but add `#![allow(clippy::disallowed_macros)]` at the top of `xtask/src/main.rs`. `xtask` is dev-only and never emits findings; nothing breaks. Alternative: per-crate `clippy.toml`s (clippy supports walking up the directory tree, so you can put a more permissive `clippy.toml` in `xtask/`). Pick the global allow for simplicity.

### The `schemars::schema_for!` macro

Returns a `schemars::Schema` (in 1.x, a `serde_json::Value` wrapper). `serde_json::to_string_pretty(&schema)` yields the file contents. The macro handles all transitive type discovery — every `JsonSchema`-deriving type reachable from `Finding` is emitted as a `$defs` entry.

### The CI sync gate

```bash
cargo run -p xtask -- gen-schema
if ! git diff --exit-code schemas/findings-v1.schema.json; then
  echo "::error::schema drifted from Rust types — re-run \`cargo xtask gen-schema\` and commit"
  exit 1
fi
```

This catches the most common failure mode: a contributor changes a Rust envelope type, forgets to regenerate the schema, opens a PR. CI says no.

## Schemars + base64-with-shape Pattern

**Decision: newtype `Base64Bytes(Vec<u8>)` with a manual `JsonSchema` impl.** Open Question 2 from CONTEXT.md is answered.

The full implementation is in Architecture Patterns → Pattern 2 above. The key insight is that **schemars 1.x's `Schema` is a thin wrapper around `serde_json::Value`**, so the manual impl can construct the schema fragment via `serde_json::json!{}` and convert. JSON Schema 2020-12's `contentEncoding` and `contentMediaType` are the standard way to describe an opaque base64 string with a known decoded media type — `application/octet-stream` is the canonical choice for "the bytes mean whatever the surrounding context says they mean" (here, an f64 array).

### Why a newtype, not a field-level `#[schemars(schema_with = ...)]`

- The newtype owns three concerns at once: serialisation (base64 encode), deserialisation (base64 decode + validation), and schema (advertise `contentEncoding`). All three travel together when the type is used as a struct field.
- A field-level `schema_with` would require a per-field paired `with = "base64_field_serde"` for serde, which duplicates the boilerplate for every usage.
- The newtype composes: `RawArray { data: Base64Bytes, shape: Vec<u64>, dtype: Dtype }` works without ceremony.

### Verification needed during Phase 1 (the spike task)

The exact schemars 1.x conversion from `serde_json::Value` into `schemars::Schema` is `try_into` per the changelog, but the precise compile shape needs verification. A 30-minute spike task in Wave 0:

1. Add `schemars = "1"` to `miner-core`.
2. Write the `Base64Bytes` newtype with the manual `JsonSchema` impl from Pattern 2.
3. `cargo run -p xtask -- gen-schema` and inspect the produced JSON.
4. Assert the generated fragment contains `"contentEncoding": "base64"`.

If the spike fails (schemars API has shifted), fall back to field-level `#[schemars(schema_with = "base64_schema_fn")]` with a paired `#[serde(with = "base64_serde")]`. Document the fallback in the PLAN.

### The `dtype` enum

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize, JsonSchema)]
#[serde(rename_all = "snake_case")]
pub enum Dtype {
    F64,
    // F32, I64, etc. can be added; v1 uses F64 only per D-01.
}
```

Single-variant enums serialise as their lone string — fine and forward-compatible.

## Stdout/Stderr Enforcement Mechanics

Three mechanisms work together:

1. **`clippy.toml` `disallowed-macros`** — see Example 2. This is the workspace-wide lint that fails CI on any direct use of `std::println` / `std::eprintln` / `std::print` / `std::eprint` / `std::dbg`. (`dbg!` is added to the list because it's the single most common accidental-stdout-pollution vector during debugging.)

2. **`StdoutSink` newtype** — see Pattern 3. The **only** place in the workspace that writes to `io::stdout()`. Even within `StdoutSink`, the writes are via `std::io::Write::write_all` and `serde_json::to_writer` — not via `println!`. This means **even `StdoutSink` does not need `#[allow(clippy::disallowed_macros)]`** because it never uses a disallowed macro. The `clippy.toml` block is purely defensive against future contributors.

3. **`tracing_subscriber::fmt().with_writer(std::io::stderr).init()`** — initialised at the top of every binary's `main()` (`miner-cli`, `miner-mcp`, `miner-http`). Every `tracing::{trace,debug,info,warn,error}!` from anywhere in the workspace routes through this subscriber to stderr. Stack traces from panics already go to stderr by default.

**Joint outcome:** stdout carries JSONL findings only. Stderr carries human-readable log lines.

## Tokio-Free Gate

```bash
# Phase 1 CI: tokio-free miner-core
set -euo pipefail

# `cargo tree -p miner-core` lists every transitive dep of miner-core.
# We're looking for: tokio (any variant), async-std, smol, async-trait macro crate,
# and any *-runtime crate that implies a runtime.
PROHIBITED='^(tokio|tokio-[^ ]+|async-std|async-std-[^ ]+|smol|smol-[^ ]+|async-trait|async-io|async-channel|async-executor|async-task)$'

LEAKED=$(cargo tree -p miner-core --edges normal,build --prefix none \
  | awk '{print $1}' \
  | sort -u \
  | grep -E "$PROHIBITED" || true)

if [ -n "$LEAKED" ]; then
  echo "::error::async runtime crate(s) leaked into miner-core:"
  echo "$LEAKED"
  exit 1
fi
echo "ok: miner-core has zero async-runtime dependencies"
```

### What's intentionally NOT in the prohibited list

- `futures-core`, `futures-channel`, `futures-task` (runtime-agnostic plumbing — fine even in `miner-core` because they describe types, not runtimes)
- `futures-util` (mostly fine; if a contributor pulls it in for combinators that's a smell but not a runtime contamination)
- `pin-project-lite` (sync code uses it sometimes)

### The `--edges normal,build` matters

`--edges normal` includes regular dependencies. `--edges build` includes `build.rs` dependencies. We deliberately *exclude* `dev` because tests are allowed to depend on async (e.g., a test could spin up a tokio runtime to verify the wrapper bridge behaviour) but production builds may not.

### What the gate does NOT catch

- A contributor adding `tokio` to `miner-reader-dukascopy` is allowed — only `miner-core` is gated.
- A contributor adding `async fn` annotations *without* a runtime crate is technically possible. `cargo clippy` does not block `async fn` per se. If this becomes a problem, add `#![forbid(async_fn_in_trait)]` to `miner-core/src/lib.rs` (verify lint name in 2024 edition).

## Config Precedence Mechanics

D-16 mandates: defaults < TOML file < env (`MINER_*`) < CLI flag, with the CLI winning.

### The figment-docs trap

The figment README shows:

```rust
Figment::new()
    .merge(Serialized::defaults(Config::parse()))   // CLI FIRST — wrong for us
    .merge(Toml::file("Config.toml"))
    .merge(Env::prefixed("APP_"))
```

This makes CLI the LOWEST precedence. Our requirement is the opposite.

### The correct order

```rust
let figment = Figment::new()
    // (no library-side defaults; missing fields are errors)
    .merge(Toml::file(resolved_toml_path))           // 1. lowest priority
    .merge(Env::prefixed("MINER_"))                  // 2.
    .merge(Serialized::defaults(cli_overrides));     // 3. highest priority (last)
let config: MinerConfig = figment.extract()?;
```

### The Option<T> + skip_serializing_if pattern

If `cli_overrides` is just `Cli::parse()` directly, the unset clap fields take their defaults — empty strings, zero ints, default `PathBuf` — and figment cannot distinguish "set to default" from "not set". The Option pattern fixes this:

```rust
// CLI overlay carries only Option<T>:
#[derive(clap::Parser, serde::Serialize)]
pub struct CliOverrides {
    #[arg(long)]
    #[serde(skip_serializing_if = "Option::is_none")]
    pub cache_root: Option<std::path::PathBuf>,

    #[arg(long)]
    #[serde(skip_serializing_if = "Option::is_none")]
    pub bar_cache_root: Option<std::path::PathBuf>,

    #[arg(long)]
    #[serde(skip_serializing_if = "Option::is_none")]
    pub output: Option<OutputDest>,
    // ...
}

// Final config has the resolved (non-optional) values:
#[derive(serde::Serialize, serde::Deserialize, JsonSchema)]
pub struct MinerConfig {
    pub cache_root: std::path::PathBuf,
    pub bar_cache_root: std::path::PathBuf,
    pub output: OutputDest,
    // ...
}
```

When `CliOverrides::cache_root` is `None`, `serde(skip_serializing_if)` omits the field entirely from the serialised form, so `Serialized::defaults` provides nothing for that field, and the env/TOML value flows through unchanged.

When `--cache-root /custom` is passed, `cache_root: Some(/custom)` serialises as `"cache_root": "/custom"`, which overrides whatever the env / TOML provided.

### Default file lookup paths

Per D-16:

```rust
// crates/miner-cli/src/config.rs (rough sketch)
fn resolve_toml_path(cli_explicit: Option<&Path>) -> Option<PathBuf> {
    if let Some(p) = cli_explicit { return Some(p.to_path_buf()); }
    // XDG: $XDG_CONFIG_HOME/miner/miner.toml || $HOME/.config/miner/miner.toml
    if let Some(proj) = directories::ProjectDirs::from("", "", "miner") {
        let xdg = proj.config_dir().join("miner.toml");
        if xdg.exists() { return Some(xdg); }
    }
    // Fallback: ./miner.toml in CWD
    let cwd = Path::new("./miner.toml");
    if cwd.exists() { return Some(cwd.to_path_buf()); }
    None
}
```

If `resolve_toml_path` returns `None`, the figment Toml provider is simply skipped (no error — config can come entirely from env + CLI).

### Required-field semantics

`MinerConfig`'s fields are non-optional. If none of TOML / env / CLI supplied a required field (e.g., `cache_root`), `figment.extract()` returns an `Err` describing the missing field. The CLI wrapper translates this into a structured stderr error per D-06 and exits with code 1 per D-07:

```rust
match figment.extract::<MinerConfig>() {
    Ok(cfg) => cfg,
    Err(e) => {
        // structured stderr JSON line:
        let err = serde_json::json!({
            "error": "missing_config",
            "details": e.to_string(),
        });
        eprintln!("{}", err);  // wait — eprintln is banned! Use the structured-stderr emitter.
        std::process::exit(1);
    }
}
```

**Caveat:** the very last `eprintln!` above is forbidden by `clippy.toml`. The right pattern is to introduce a tiny `stderr_emit` helper in `miner-core` (parallel to `StdoutSink`) that writes structured-error JSON to stderr via `io::Write` — never via the `eprintln!` macro. That helper is the **only** writer to `io::stderr()` for preflight errors, and it can carry the `#[allow]` if needed. The tracing-subscriber is for log events; the structured-stderr-error emitter is for one-shot pre-flight rejection messages (D-06). These are semantically distinct.

Plan for Phase 1: introduce both `findings::sink::StdoutSink` and `error::stderr_emit::write_preflight_error(&PreflightError)` — they're sibling concepts.

## error_code Vocabulary

Phase 1 must lock the vocabulary now to keep the enum stable (per CONTEXT.md open question 3). The codes below are split into the two contexts D-05 / D-06 distinguish: mid-stream `scan_error` codes and pre-flight error codes.

### Pre-flight `error_code` strings (D-06; emitted to stderr before any work)

| Code | When emitted | Example details |
|------|--------------|-----------------|
| `invalid_parameter` | A CLI/MCP/HTTP parameter fails type / range / enum validation. | `{"error":"invalid_parameter","parameter":"timeframe","got":"1m","allowed":["15m","1h","1d"]}` |
| `unknown_scan` | The requested `scan_id@version` is not in the registry. | `{"error":"unknown_scan","scan_id":"made.up.scan","version":1}` |
| `unknown_instrument` | The requested instrument is not in the source catalog. | `{"error":"unknown_instrument","symbol":"FAKEUSD"}` |
| `missing_required_config` | A required `MinerConfig` field could not be resolved from any layer. | `{"error":"missing_required_config","field":"cache_root"}` |
| `invalid_config` | Config file/env value failed parse or type-check (TOML invalid, env var not parseable, etc.). | `{"error":"invalid_config","field":"cache_root","reason":"path does not parse as UTF-8"}` |
| `sweep_too_large` | Sweep cardinality exceeds a configured upper bound (per PITFALLS §12). Reserved for Phase 5 sweep runner. | `{"error":"sweep_too_large","cardinality":4200000000,"limit":100000000}` |
| `internal_error` | Catastrophic failure unrelated to inputs (panic boundary caught, disk full at startup, etc.). Exit code 1. | `{"error":"internal_error","message":"..."}` |

### Mid-stream `error_code` strings (D-05; emitted inside `kind: scan_error` findings)

| Code | When emitted | Notes |
|------|--------------|-------|
| `coverage_gap` | A scan discovered a gap manifest mid-run that the gap policy disallowed. (Strict policy is supposed to fail pre-flight, but this code exists for `continuous_only` cases that turn unrecoverable.) Phase 3+ uses primarily. | |
| `compute_error` | A scan's compute kernel failed (numerical issue, NaN propagation rejected, etc.). | Generic catch-all for scan-internal compute failures. Phase 4+ scans populate. |
| `cache_corruption` | Derived-bar cache read failed checksum/version; partial-rebuild path engaged. Phase 2+. | |
| `internal_panic_caught` | A panic was caught at the scan boundary. Should be vanishingly rare. | The `message` field carries the panic payload (stringified). |

### Why this set

- The Phase 6 roadmap mentions `invalid_parameter`, `coverage_gap`, `sweep_too_large` explicitly — all three are reserved here.
- The set is **closed for pre-flight** (every Phase 1+ pre-flight rejection has a code) and **forward-extensible for mid-stream** (new scans in Phase 4+ may invent compute-specific codes, but they all fall under `compute_error` until proven otherwise — the `message` carries the per-scan specifics).
- Schema-wise, `error_code` is a `string` in the JSON Schema (not an `enum`) — this keeps the schema additive-only when new codes appear later (no breaking change). The Rust side may still use a typed `enum ErrorCode` internally and `Serialize` it as the snake_case string.

### Concrete Rust shape (Phase 1)

```rust
// crates/miner-core/src/error/codes.rs
#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize, JsonSchema)]
#[serde(rename_all = "snake_case")]
pub enum PreflightCode {
    InvalidParameter,
    UnknownScan,
    UnknownInstrument,
    MissingRequiredConfig,
    InvalidConfig,
    SweepTooLarge,
    InternalError,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize, JsonSchema)]
#[serde(rename_all = "snake_case")]
pub enum ScanErrorCode {
    CoverageGap,
    ComputeError,
    CacheCorruption,
    InternalPanicCaught,
}
```

Both enums get `JsonSchema` derive. The findings envelope's `error_code` field has type `String` (open string) in the public schema even if internally a `ScanErrorCode` is serialised to it — this is the additive-extensibility property.

## run_start / run_end Framing Mechanics

D-09 + D-10 + D-11 specify what must happen. The mechanics:

### ULID generation

```rust
// crates/miner-core/src/findings/run_id.rs
use ulid::Ulid;
use serde::{Serialize, Deserialize};
use schemars::JsonSchema;

#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, Serialize, Deserialize)]
#[serde(transparent)]
pub struct RunId(Ulid);

impl RunId {
    pub fn new() -> Self { Self(Ulid::new()) }
}

impl JsonSchema for RunId {
    fn schema_name() -> std::borrow::Cow<'static, str> { "RunId".into() }
    fn schema_id() -> std::borrow::Cow<'static, str> { "miner_core::findings::RunId".into() }
    fn json_schema(_: &mut schemars::SchemaGenerator) -> schemars::Schema {
        serde_json::json!({
            "type": "string",
            "pattern": "^[0-9A-HJKMNP-TV-Z]{26}$",
            "description": "26-character Crockford-base32 ULID, time-prefixed and lexicographically sortable"
        }).try_into().expect("valid schema fragment")
    }
}
```

`ulid::Ulid::new()` is the canonical generation API. `serde(transparent)` makes the JSON form just the inner string. The custom `JsonSchema` makes the schema strict (only Crockford-base32 chars, exactly 26 of them).

### `started_at_utc` population

`chrono::Utc::now()` returns a `DateTime<Utc>`. With the chrono `serde` feature, the default serialisation is RFC3339 (e.g., `"2026-05-15T14:33:21.412Z"`).

```rust
let started = chrono::Utc::now();   // captured once at run_start emission
// ... work ...
let ended = chrono::Utc::now();
let wall_clock_ms = ended.signed_duration_since(started).num_milliseconds();
```

(Note: `signed_duration_since().num_milliseconds()` is `i64`. For sub-millisecond rounding behaviour, prefer `chrono::Duration` arithmetic.)

### Summary record assembly

Phase 1 has no scans, but the data shape is fixed by D-11:

```rust
#[derive(Debug, Clone, Default, Serialize, Deserialize, JsonSchema)]
pub struct RunSummary {
    pub results_emitted: u64,
    pub scan_errors: u64,
    pub gap_aborted: u64,
    // Keyed by "scan_id@version" string; BTreeMap for deterministic ordering (OUT-03 lays groundwork)
    pub per_scan: std::collections::BTreeMap<String, PerScanCounts>,
}

#[derive(Debug, Clone, Copy, Default, Serialize, Deserialize, JsonSchema)]
pub struct PerScanCounts {
    pub results: u64,
    pub errors: u64,
    pub gap_aborted: u64,
}
```

Phase 1's `emit-fixture` CLI subcommand emits `RunSummary::default()` inside `RunEnd`. Real scans (Phase 3+) maintain running counters and emit the populated summary.

### Where the counts live

Phase 1 should reserve the counter shape but NOT implement counting (no scans run). Phase 3 (when the scan engine appears) is the right place to wire counters into the engine's `run_scan` orchestration. Phase 1's job: define the types so Phase 3 doesn't reshape the envelope.

## Open Risks & Crate-Version Notes

These are surfaced for the planner / plan-checker to weigh, not for the user to re-litigate.

### Risk 1: `resolver = "2"` vs `"3"` (D-20 drift)

**Severity:** LOW.

CONTEXT.md D-20 says `resolver = "2"`. Edition 2024 *implies* `resolver = "3"` (Rust 1.85 release notes). Setting both `edition = "2024"` and `resolver = "2"` does not error but silently disables the MSRV-aware resolver. The fix is one-character (`"3"` not `"2"`) and aligns with the user's clear intent ("most pragmatic and/or popular options Rust community members would recognise").

**Recommendation:** Write `resolver = "3"` in the workspace Cargo.toml. Document the divergence in PLAN.md Risks as a non-substantive correction.

### Risk 2: `schemars` 1.x API surface for manual `JsonSchema` impls

**Severity:** LOW-MEDIUM.

`schemars` 1.0 collapsed `Schema` to a `serde_json::Value` wrapper. The manual `JsonSchema` impls in Patterns 2 and "run_start framing" rely on `try_into` from `serde_json::Value`. This is consistent with the 1.x changelog but the exact compile shape was not verified in this session.

**Recommendation:** First task in Phase 1 Wave 0 is a half-day spike that compiles `Base64Bytes` + `RunId` + a stub `Finding` and runs `cargo xtask gen-schema`. If the spike fails on the `try_into`, the fallback is `Schema::new(serde_json::json!{...})` or whatever the actual constructor is — adjust the patterns once verified. **Do not write the rest of the envelope before the spike succeeds.**

### Risk 3: `thiserror` + `serde::Serialize` derive does not compose for `io::Error`

**Severity:** LOW (well-documented workaround exists).

If `MinerError::Io(#[from] std::io::Error)` is in the enum and we slap `#[derive(Serialize)]` on, the build breaks because `std::io::Error: !Serialize`.

**Recommendation:** Split the error model into (a) internal `MinerError` with `thiserror` derive only, and (b) wire-level `WireError { code: PreflightCode | ScanErrorCode, message: String, context: BTreeMap<String, serde_json::Value> }` with `serde::Serialize`. The conversion is a small function that calls `err.to_string()` for `message`. This matches D-18's intent ("Library errors implement `serde::Serialize` so they can be embedded inside `kind: scan_error` envelopes") without fighting the type system.

### Risk 4: `rmcp` cliffs surfacing in Phase 1 (early-warning only)

**Severity:** N/A for Phase 1 (rmcp does not appear). LOW probability of surfacing.

`rmcp` is the Phase 6 dependency, current version 0.16.0. It is a v0.x crate (breaking-change-allowed-by-semver) and is the highest-risk dep in the project. *Phase 1 does not depend on rmcp* — `miner-mcp` is an empty crate per D-23-style reasoning. However, two things could leak rmcp concern into Phase 1:

1. If a contributor adds `rmcp` to the workspace dependency table proactively (anticipating Phase 6), that pulls tokio into the workspace `Cargo.lock`. Tokio in the lockfile is fine; tokio as a dep of `miner-core` is not (gate catches this).
2. The empty `miner-mcp/src/main.rs` should not import rmcp. Phase 6 wires it in.

**Recommendation:** Phase 1 plan should explicitly say "miner-mcp/Cargo.toml has zero dependencies in Phase 1 (placeholder `fn main() {}`); rmcp is added in Phase 6". A grep for `rmcp` in Phase 1 CI would catch any premature addition.

### Risk 5: Documentation of "the spike that wasn't run"

**Severity:** LOW.

Multiple places in this research recommend "spike tasks." The PLAN must call these out as explicit named tasks in Wave 0 so they don't get folded into other work and silently skipped. The two required spikes are:

1. **schemars 1.x base64-with-shape spike** — half-day; compile `Base64Bytes`, run `xtask gen-schema`, validate output.
2. **figment + clap precedence spike** — quarter-day; build a tiny CliOverrides + figment pipeline, set `MINER_CACHE_ROOT=/env` + `--cache-root /cli` + nothing in TOML, assert result is `/cli`; flip flags off in various combinations, assert the right value flows through.

If either spike returns a surprise, the plan branches to the documented fallback (manual `JsonSchema` field attributes for the first; possibly a switch to `config` crate or a hand-rolled layered config for the second — though we expect figment to work as analysed).

### Risk 6: `chrono` deprecation drift

**Severity:** LOW.

`chrono` 0.4.x is mature but has had RustSec advisories in older versions and competing crates (`jiff`, `time`) are gaining mindshare. D-17 locks `chrono`. Phase 1 should pin a recent 0.4.x and `cargo audit` should be set up (Phase 7 owns the gate, but Phase 1 can ship a passive `cargo audit` CI step that warns without failing).

### Risk 7: Empty crates causing cargo warnings

**Severity:** LOW.

`miner-reader-dukascopy`, `miner-mcp`, `miner-http`, `miner-bench` are empty in Phase 1. An empty `src/lib.rs` or `src/main.rs` compiles fine but the lib crates need at least `pub fn _placeholder() {}` or a `#[allow(unused)]` annotation to avoid clippy "empty crate" noise. The bin crates need a minimal `fn main() {}`. Per D-23 the bench crate's `fn main()` cannot use `println!` — use a `tracing::info!` (which initialises the subscriber in main first).

**Recommendation:** Each empty crate ships with:

```rust
//! Phase N: implementation forthcoming.
// (lib crate)
pub fn _placeholder() {}
// (bin crate)
fn main() {
    tracing_subscriber::fmt().with_writer(std::io::stderr).init();
    tracing::info!("placeholder binary; real implementation in Phase N");
}
```
