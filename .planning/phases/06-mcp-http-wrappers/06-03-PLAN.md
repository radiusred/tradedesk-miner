Canonical CLI invocation forms (from crates/miner-cli/src/cli.rs):
  miner scan <scan_id@version> --instrument SYMBOL:side [--instrument SYMBOL:side] \
              --timeframe <15m|1h|1d> --window <YYYY-MM-DD:YYYY-MM-DD> \
              [--param key=value]... [--gap-policy strict|continuous_only] \
              [--seed <u64>] [--bootstrap stationary|block] [--bootstrap-n <u32>] \
              [--null circular_shift|iaaft] [--null-n <u32>] [--dry-run]
  miner sweep <manifest.toml> [--dry-run] [--seed <u64>]
  miner scans                  # JSONL catalogue introspection
  miner emit-fixture <variant> # Reserved for golden fixture generation

Four-tier exit codes (D3-24):
  0   — clean run; findings emitted; RunEnd reached.
  1   — preflight or kernel error; structured WireError emitted on stderr or as a ScanError envelope; non-empty stdout possible if some Result envelopes already streamed.
  2   — invalid usage / clap-derive arg parse failure; no envelopes emitted.
  130 — SIGINT (POSIX convention 128+2); RunEnd may be emitted carrying any findings already streamed; partial output considered valid.

    1. `# Future MCP + HTTP wrappers (deferred to v2)` — H1.
    2. Intro paragraph: miner's v1 surface is the CLI. The MCP and HTTP server wrappers — originally scoped for Phase 6 — have been deferred to v2. Rationale: miner's primary consumer (the RadiusRed Quant agent) runs subprocess + JSONL just fine; running a 24x7 server for a single-operator tool is not desired in v1.     3. `## What MCP would expose` — Architectural surface, not contract:
       - One MCP tool per scan (`stats.autocorr.ljung_box@1`, `stats.stationarity.adf@1`, ...). Parameter schemas derive from the existing registry. Tool result is the streamed JSONL of `Finding` envelopes.
       - Meta-tools: `list_scans` (mirrors `miner scans`), `list_symbols` (lists instrument-side pairs available in the configured cache), `probe` (preflight a parameter set without running).
       - Transport: stdio for local-agent attachment, streamable-HTTP for remote agents.
       - One open design question for v2 plan-phase: one-tool-per-scan vs a single generic `scan` tool that takes a `scan_id@version` parameter. v1 has no opinion.
       - Deep design: `.planning/research/ARCHITECTURE.md §8` (Wrapper Crates).
    4. `## What HTTP would expose` — Same shape:
       - Routes: `GET /v1/scans` (catalogue), `GET /v1/symbols` (instrument inventory), `POST /v1/scan` (single scan; streams findings as response body), `POST /v1/sweep` (sweep manifest; same).
       - Content negotiation: NDJSON (default) or SSE (when the client prefers a framed event stream). Open question for v2.
       - Authentication, body cap, rate limits, mTLS, bind address — out of v1 sketch scope; v2 owns these.
       - Deep design: same source.
    5. `## Crate choices` — Source: `.planning/research/STACK.md`:
       - **MCP**: `rmcp` (the official-ish modelcontextprotocol/rust-sdk Rust crate). Confidence MEDIUM in STACK.md (marked `VERIFY` — version, transport support, streaming tool-result chunks, and tokio compatibility need re-confirming when v2 plan-phase begins). Fallback if `rmcp` doesn't fit: hand-rolled JSON-RPC-over-stdio (~500 LOC against `serde_json`; documented as the safe path). The fallback does NOT affect the HTTP wrapper.
       - **HTTP**: `axum` 0.7+ over `tokio` 1.40+ with `tower-http` middleware (tracing + compression + timeouts). Confidence HIGH.
       - **Async bridge**: `tokio::task::spawn_blocking` is the canonical pattern. `miner-core` stays sync + rayon; the wrappers offload `engine::run_one` / `sweep::run_sweep` invocations into the blocking pool so the async runtime workers stay non-blocking. FOUND-04 (`cargo tree -p miner-core --edges normal,build` must continue to show zero async deps) is non-negotiable for v2.
    6. `## Why deferred` — Three short bullets:
       - The CLI is enough for v1 — miner's primary consumer (the Quant agent) runs subprocess + JSONL with no shape gap.
       - Running a 24x7 server adds operational surface (deploy, monitor, restart, secret rotation) for a single-operator tool.
       - `rmcp` is the highest-risk single dependency in the stack; better to let v2 plan-phase re-research it against the then-current SDK release than lock the choice now.

          """Decode one RawArray dict {dtype, shape, b64} into a numpy ndarray."""
          dtype = np.dtype(raw_array["dtype"])
          shape = tuple(raw_array["shape"])
