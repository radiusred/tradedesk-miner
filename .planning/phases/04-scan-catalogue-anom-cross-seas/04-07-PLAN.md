    - `pub struct OlsWindowResults { pub betas: Vec<f64>, pub alphas: Vec<f64>, pub r2s: Vec<f64>, pub residual_stds: Vec<f64> }`
    - `#[inline] pub(super) fn rolling_ols(y: &[f64], x: &[f64], window: usize) -> OlsWindowResults` — for each i in 0..n-window+1:
      1. Build a `nalgebra::SMatrix<f64, _, 2>` design matrix X (rows = window, cols = 2: intercept column of 1s + x slice)
      2. Build a `nalgebra::SVector<f64, _>` response y slice — note: nalgebra requires compile-time-fixed sizes for SMatrix; the rolling window size is a runtime param so use `nalgebra::DMatrix` instead, OR use a fixed-window-size SMatrix template that costs zero heap. Plan choice (per RESEARCH.md §Alternatives Considered note "small fixed sizes"): use nalgebra::DMatrix for the design (runtime-sized) — the matrices are small (window × 2) and DMatrix is still heap-efficient for these dimensions. Document this choice in a kernel doc comment.
      3. Solve (X.tranpose() * X).try_inverse().unwrap() * X.transpose() * y to get coefficients [α, β]
      4. Compute residuals = y - X * coefficients
      5. R² = 1 - SS_res / SS_tot where SS_res = residuals.dot(&residuals); SS_tot = (y - y_mean).dot(&(y - y_mean))
      6. residual_std = sqrt(SS_res / (window - 2))
    - debug_assert!(window >= 3, "rolling_ols: need window ≥ 3 for residual_std df")
