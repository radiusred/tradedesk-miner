key-decisions:
  - "SweepOptions.sleep_after_first_finding_ms field is ungated
    (always present), but the executor's propagation into per-job
    ScanRequest.sleep_after_first_finding_ms IS cfg-gated to
    test/test-internal. Rationale: ScanRequest's own field is
    cfg-gated, so cross-cfg struct literals would otherwise need
    cfg branches. Keeping SweepOptions field uniform across all
    feature sets gives the CLI a single `SweepOptions { .. }`
    construction call site (with the value set under cfg) and
    keeps the cfg complexity localised inside the executor — a
    cleaner boundary than threading `#[cfg]` through every
    `SweepOptions { .. }` literal in production + test code."
  - "Universal hygiene flags on `miner scan` mirror the manifest's
    [hygiene] block grammar — same accepted values (`stationary` /
    `block` for bootstrap; `phase_scramble` / `circular_shift` for
    null), same engine-side enum dispatch (`BootstrapMethod` /
    `NullMethod`), same error code on unknown strings
    (`PreflightCode::InvalidParameter`). The CLI surface is a thin
    parsed-layer over `manifest.rs::parse_*_method` — duplicated
    helpers in `scan_args.rs` rather than re-exported from miner-core
    so the CLI keeps its own boundary error type (`WireError`) and
    the manifest helpers stay String-returning (for the structured
    `with_context` enrichment in `manifest::validate`)."
  - "tests/REFERENCE-VERSIONS.md lives at workspace root (NOT under
    crates/miner-core/tests/goldens/) — the per-crate file pins
    Python statsmodels/scipy/pandas for Phase 4 goldens; the new
    workspace-root file pins R 4.4.x + tseries + stats for Phase 5
    BH-FDR + Politis-White goldens. Keeping them separate preserves
    the per-crate scoping (Phase 4 goldens live in miner-core; Phase
    5 R-toolchain goldens are TBD across crates) and matches the
    plan's literal `tests/REFERENCE-VERSIONS.md` path."
  - "`#[file_serial(miner_bin_test_internal)]` (cross-process) instead
    of `#[serial]` (in-process) — `cargo test -p miner-cli` runs each
    `tests/*.rs` file as a separate process. Three of those tests
    (sigint_preserves_stream, sigint_mid_sweep,
    cancel_overrides_error_exit_130) all invoke
    `cargo build -p miner-cli --features test-internal --bin miner`
    inside their first step, racing to rebuild the shared
    `target/debug/miner` artifact. `#[serial]` alone doesn't help
    here because each test file is a separate process. Adding the
    `file_locks` feature to `serial_test` and switching to
    `#[file_serial(token)]` synchronises all three tests on a single
    cross-process fslock-backed mutex."
  - "`miner sweep --help` includes the global flags interleaved with
    the sweep-specific flags because clap renders global flags
    (defined on `Cli`) on every subcommand. The output is a
    superset of the documented sweep-args surface — verified by
    inspection but not pinned by a regression test (Phase 6's
    --help-snapshot test will pin the wire form across all
    subcommands when MCP/HTTP wrappers ship)."

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] `SweepOptions` field needed to be ungated
to keep cross-feature-set struct-literal construction simple.**

- **Found during:** Task 3 sigint_mid_sweep build (the `miner` binary
  built without `test-internal` triggered E0063 on the
  `SweepOptions { dry_run, sleep_after_first_finding_ms }` struct
  literal — when the field was `#[cfg(any(test, feature = "test-internal"))]`-gated
  in `miner-core`, the CLI's main.rs struct literal couldn't compile
  uniformly across feature sets, because the gate had to fire on
  miner-core's feature, not the CLI's).
- **Issue:** `cfg(any(test, feature = "test-internal"))` checked
  on the CLI's own feature, not miner-core's. Without
  `--features test-internal` on miner-cli, the gate evaluates false
  even though miner-core had `test-internal` enabled (via dev-deps).
  The result was a missing-field E0063 on the SweepOptions literal.
- **Fix:** Promote `sleep_after_first_finding_ms: Option<u64>` to
  be an unconditional field on `SweepOptions`. The cfg gate moves
  to the executor's propagation (`r.sleep_after_first_finding_ms = …`
  is gated; the field exists, but the downstream ScanRequest field
  is also gated, so release builds never see the value plumb through).
  Effectively the field is always present-but-inert at the boundary;
  the test-only behaviour boundary lives inside miner-core's executor
  + kernel.
- **Files modified:** `crates/miner-core/src/sweep/executor.rs`,
  `crates/miner-cli/src/main.rs`,
  `crates/miner-core/tests/sweep_dry_run.rs` (gained
  `..Default::default()` on the SweepOptions struct literal).
- **Verification:** both `cargo build --bin miner` (no features) and
  `cargo build --features test-internal --bin miner` compile;
  `./target/release/miner sweep --help` does NOT show
  `--sleep-after-first-finding-ms` (cfg gate on ScanArgs / SweepArgs
  still works correctly).
- **Committed in:** `09a1dbb` (Task 3 feat-cum-test).

**2. [Rule 1 — Bug] sigint_mid_sweep test failed when run alongside
sigint_preserves_stream because both rebuilt `target/debug/miner`
with `--features test-internal` from separate processes.**

- **Found during:** Task 3 — first full `cargo test -p miner-cli`
  run after writing the new tests.
- **Issue:** `cargo test -p miner-cli` runs each `tests/*.rs` file
  in its own process; `#[serial_test::serial]` only synchronises
  tests within a single process. Three tests (the two SIGINT tests
  + cancel_overrides_error_exit_130) each call
  `cargo build -p miner-cli --features test-internal --bin miner`
  as their first step. Concurrent processes raced to rebuild
  `target/debug/miner`; the race surfaced as the sweep_summary
  envelope unexpectedly persisting in the captured stdout (the
  test was reading from a binary built without the sleep hook,
  so the kernel never paused, so par_iter completed BOTH jobs
  before SIGINT had a chance to set cancel).
- **Fix:** Enable `serial_test`'s `file_locks` feature in
  `crates/miner-cli/Cargo.toml`, and swap the three tests'
  `#[serial_test::serial]` attribute for
  `#[serial_test::file_serial(miner_bin_test_internal)]`. The
  `file_serial` macro uses an fslock-backed mutex that crosses
  process boundaries.
- **Files modified:** `crates/miner-cli/Cargo.toml`,
  `crates/miner-cli/tests/sigint_preserves_stream.rs`,
  `crates/miner-cli/tests/sigint_mid_sweep.rs`,
  `crates/miner-cli/tests/cancel_overrides_error_exit_130.rs`.
- **Verification:** `cargo test -p miner-cli` runs the full suite
  green (8 test binaries; 53 integration tests); two consecutive
  runs both pass. `cargo test --workspace --all-features` exits 0.
- **Committed in:** `09a1dbb` (Task 3).

**3. [Rule 2 — Clippy hygiene] Doc-comment backtick lints on
`InvalidParameter` / `WireError` / `SyncCache` / `SweepSummary` /
etc. in test files.**

- **Found during:** Task 1, Task 3 clippy checks.
- **Issue:** Standard workspace `-D warnings` gate fires
  `clippy::doc_markdown` on every API identifier that lacks
  backticks in a doc-comment — including identifiers in
  test-internal docstrings that are descriptive prose, not API
  documentation.
- **Fix:** Per-file `#![allow(clippy::doc_markdown, reason = "test
  docstrings are descriptive prose, not API identifiers")]` on the
  two new integration-test files; per-line backtick fixes on the
  three unit-test doc-comments in `scan_args.rs` and `sweep_args.rs`.
- **Commits:** `5a3cf82` + `09a1dbb`.
